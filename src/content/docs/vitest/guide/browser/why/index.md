# 브라우저 모드가 필요한 이유

## 동기

우리는 테스트 워크플로를 개선하고 더 정확하고 신뢰할 수 있는 테스트 결과를 얻기 위해 Vitest 브라우저 모드 기능을 개발했습니다. 테스트 API에 추가된 이 기능을 통해 개발자는 네이티브 브라우저 환경에서 테스트를 실행할 수 있습니다. 이 섹션에서는 이 기능의 도입 배경과 테스트 측면의 이점을 살펴봅니다.

### 다양한 테스트 방식

JavaScript 코드를 테스트하는 방법은 여러 가지가 있습니다. 일부 테스트 프레임워크는 Node.js에서 브라우저 환경을 시뮬레이션하고, 다른 프레임워크는 실제 브라우저에서 테스트를 실행합니다. 이 맥락에서 [jsdom](https://www.npmjs.com/package/jsdom)은 Jest 또는 Vitest 같은 테스트 러너와 함께 사용되어 브라우저 환경을 시뮬레이션하는 spec 구현의 예시입니다. 반면 [WebdriverIO](https://webdriver.io/)나 [Cypress](https://www.cypress.io/) 같은 다른 테스트 도구는 실제 브라우저에서 애플리케이션을 테스트할 수 있게 해주며, [Playwright](https://playwright.dev/)의 경우 브라우저 엔진을 제공합니다.

### 시뮬레이션의 한계

jsdom 또는 happy-dom 같은 시뮬레이션 환경에서 JavaScript 프로그램을 테스트하면 테스트 설정이 단순해지고 사용하기 쉬운 API를 제공하므로 많은 프로젝트에 적합하며 테스트 결과에 대한 신뢰도를 높여줍니다. 하지만 이러한 도구는 실제 브라우저가 아니라 브라우저 환경을 시뮬레이션할 뿐이라는 점을 반드시 염두에 두어야 하며, 이로 인해 시뮬레이션 환경과 실제 환경 사이에 차이가 발생할 수 있습니다. 따라서 테스트 결과에서 false positive 또는 false negative가 발생할 수 있습니다.

테스트에 대해 가장 높은 수준의 신뢰를 확보하려면 실제 브라우저 환경에서 테스트하는 것이 매우 중요합니다. 이것이 우리가 Vitest에 브라우저 모드 기능을 개발한 이유이며, 이를 통해 개발자는 브라우저에서 테스트를 네이티브하게 실행하고 더 정확하고 신뢰할 수 있는 테스트 결과를 얻을 수 있습니다. 브라우저 수준 테스트를 통해 개발자는 실제 환경에서도 애플리케이션이 의도한 대로 동작할 것이라는 확신을 더 가질 수 있습니다.

## 단점

Vitest 브라우저를 사용할 때는 다음과 같은 단점을 고려해야 합니다.

### 초기 개발 단계

Vitest의 브라우저 모드 기능은 아직 개발 초기 단계에 있습니다. 따라서 아직 완전히 최적화되지 않았을 수 있으며, 아직 해결되지 않은 버그나 이슈가 있을 수 있습니다. 사용자는 WebdriverIO, Cypress, Playwright 같은 독립형 브라우저 측 테스트 러너를 함께 사용해 Vitest 브라우저 사용 경험을 보완하는 것이 권장됩니다.

### 더 긴 초기화 시간

Vitest 브라우저는 초기화 과정에서 provider와 브라우저를 구동해야 하므로 시간이 다소 걸릴 수 있습니다. 이로 인해 다른 테스트 패턴과 비교해 초기화 시간이 더 길어질 수 있습니다.
